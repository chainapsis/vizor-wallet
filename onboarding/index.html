1. Architecture Overview

1.1 Layer Separation

The wallet is split into three distinct layers:

LayerLanguageResponsibilityKey Files
Rust Core Rust All Zcash cryptography: key derivation (ZIP-32), address encoding (ZIP-316), block scanning (trial decryption), wallet DB (SQLite), commitment tree management rust/src/wallet/, rust/src/api/

1.3 Why Rust + Flutter

The official Zcash ecosystem provides mobile SDKs for Swift (iOS) and Kotlin (Android), both of which wrap the same librustzcash Rust crates via UniFFI / JNI. There is no official Flutter/Dart SDK.

Our approach calls librustzcash directly via FFI, which gives us:

  • Single Rust codebase for both iOS and Android — no platform-specific crypto code
  • Direct access to zcash_client_backend and zcash_client_sqlite without an intermediary SDK
  • flutter_rust_bridge handles cross-compilation, async, and type marshalling automatically

2. Flutter Rust Bridge (FRB) Communication

2.1 How FRB Works

FRB reads Rust source files in rust/src/api/ and generates Dart bindings. At runtime:

  1. Dart calls a generated async function (e.g. createWallet())
  2. FRB serializes arguments using SSE (Simple Serialization Engine)
  3. The call is dispatched to a Rust function on a separate isolate/thread
  4. Rust executes the function, serializes the result
  5. FRB deserializes the result back into Dart objects

Rust functions marked #[flutter_rust_bridge::frb(sync)] run synchronously on the calling thread (no isolate hop). Use this only for trivial, non-blocking operations.

2.3 Codegen Workflow

# (adding/removing/changing pub fn signatures or pub structs)
flutter_rust_bridge_codegen generate

# This regenerates:
#   lib/src/rust/api/*.dart        (Dart bindings)
#   lib/src/rust/frb_generated.*   (FFI glue)
#   rust/src/frb_generated.rs      (Rust FFI glue)

Important: Run codegen from the project root (/zcash-wallet/), not from rust/. The flutter_rust_bridge.yaml config is in the project root.

2.4 Design Constraints

  • Simple types only at the API boundary. FRB handles primitives, String, Vec, and flat structs well. Complex Zcash types (WalletDb, UnifiedSpendingKey, CompactBlock) must stay inside Rust. The API layer (rust/src/api/) converts between simple types and internal Zcash types.
  • No persistent Rust state across calls. Each FFI call opens the database, performs the operation, and closes. There is no long-lived WalletDb instance held in Rust. This is simpler and avoids lifecycle issues.
  • Errors are strings. Result<T, String> is used throughout the API. Zcash crate errors are formatted into human-readable strings at the API boundary.

3.2 Wallet API — Key Derivation & Account Management

File: rust/src/wallet/keys.rs

Mnemonic generation: Uses bip0039 crate to generate 24-word BIP-39 mnemonics with 256-bit entropy. Seed derivation follows the standard: PBKDF2-HMAC-SHA512 with salt "mnemonic" + passphrase (empty passphrase).

Key derivation: zcash_keys::UnifiedSpendingKey::from_seed() derives three key types from a single seed:

  • Transparent: BIP-44 path m/44'/133'/0' (mainnet)
  • Sapling: ZIP-32 HD derivation on the Jubjub curve
  • Orchard: ZIP-32 HD derivation on the Pallas curve

Account creation: WalletDb::create_account() stores only the Unified Full Viewing Key (UFVK) in the database. The spending key is never persisted in the DB.

Address encoding: The Unified Address is generated with Require Orchard + Require Sapling + Omit Transparent, matching zodl/Zashi wallet behavior. This is controlled by the shielded_address_request() helper function.

fn shielded_address_request() -> UnifiedAddressRequest {
    UnifiedAddressRequest::custom(
        ReceiverRequirement::Require, // Orchard
        ReceiverRequirement::Require, // Sapling
        ReceiverRequirement::Omit,    // Transparent
    ).expect("valid receiver requirements")
}

File: rust/src/wallet/sync.rs

Individual sync steps exposed to Dart. Each function opens the database, performs one operation, and returns. Dart orchestrates the sequence.

3.4 Zcash Crate Dependency Graph

zcash_client_sqlite (0.19.5) <-- Our main entry point |-- zcash_client_backend (0.21.2) | |-- zcash_keys (0.12.0) | |-- zcash_protocol (0.7.2) | |-- zcash_address (0.10.1) | |-- zcash_primitives (0.26.4) | +-- lightwalletd proto (via tonic-prost) |-- sapling-crypto (0.5.x) <-- Must match zcash_client_sqlite's version +-- orchard (0.11.x) <-- Must match zcash_client_sqlite's version

Version constraint: orchard and sapling-crypto versions must match what zcash_client_sqlite uses internally. If you use different versions, you get type mismatch errors (e.g. "expected sapling_crypto::Node, found Node") because Rust treats different crate versions as different types.

3.5 Security Model: Spending Key Separation

This is an intentional design by librustzcash, not our choice:

  • WalletDb stores only the Unified Full Viewing Key (UFVK) — sufficient for detecting transactions and computing balances, but cannot spend funds.
  • The mnemonic / seed is stored in Dart's flutter_secure_storage (iOS Keychain / Android Keystore), encrypted at rest by the OS.
  • The spending key is derived on-the-fly from the seed only when needed for transaction signing, passed to Rust via FFI, and used with secrecy::SecretVec for memory zeroization after use.

4. Dart / Flutter Layer

4.1 Riverpod Providers

walletProvider (AsyncNotifierProvider<WalletNotifier, WalletState>)

  • On build(): checks if wallet DB exists, loads address from UFVK
  • createWallet(): generates mnemonic, creates account, saves mnemonic to secure storage
  • importWallet(): validates mnemonic, deletes existing DB, creates account
  • WalletState: { hasWallet, unifiedAddress, network }

syncProvider (AsyncNotifierProvider<SyncNotifier, SyncState>)

  • Manages the entire sync lifecycle via gRPC + Rust FFI
  • startSync(): connects to lightwalletd, runs the sync loop
  • stopSync(): sets cancellation flag
  • SyncState: { isSyncing, percentage, scannedHeight, chainTipHeight, transparentBalance, saplingBalance, orchardBalance, totalBalance, error }

4.2 Sync Loop Orchestration

The sync loop runs in Dart (not Rust), following the zodl/Zashi Android SDK pattern where Kotlin's CompactBlockProcessor manages the loop. Our equivalent is SyncNotifier._runSync().

// Simplified sync loop (actual code in sync_provider.dart)
1. Connect gRPC to zec.rocks:443
2. Get chain tip height
3. Call Rust: update_chain_tip(height)
4. Download Sapling + Orchard subtree roots via gRPC stream
5. Call Rust: put_subtree_roots(sapling_roots, orchard_roots)
6. Loop:
   a. Call Rust: suggest_scan_ranges() -> ranges
   b. If no ranges: done (fully synced)
   c. Pick highest-priority range, batch = 1000 blocks
   d. Download compact blocks via gRPC stream
   e. Write block files to {cache}/blocks/{height}-{hash}-compactblock
   f. Call Rust: write_block_metadata(metas)
   g. Get tree state via gRPC: getTreeState(start - 1)
   h. Call Rust: scan_blocks(from_height, tree_state, limit)
   i. Call Rust: get_sync_status() + get_balance()
   j. Update SyncState -> UI refreshes

5. Sync Engine Deep Dive

5.1 Full Sync Flow

Initial sync on mainnet starts from Sapling activation height (419,200) and must scan through to the current tip (~3.29M blocks). This takes hours on mobile.

The scan range priority system determines what to scan first:

PriorityValueMeaning
Verify6Chain reorg detection — scan immediately
ChainTip5Latest blocks — shows recent transactions
FoundNote4Complete commitment trees for found notes
OpenAdjacent3Near account birthday
Historic2Advance fully-scanned height
Scanned1Already scanned (skipped)
Ignored0Not relevant (skipped)

5.2 Block Cache & FsBlockDb

FsBlockDb from zcash_client_sqlite is a filesystem-backed block cache:

  • Metadata DB: {cache_path}/blockmeta.sqlite — tracks which blocks are cached
  • Block files: {cache_path}/blocks/{height}-{hash_hex}-compactblock
  • Hash in filename is byte-reversed hex (matches BlockHash::Display)
  • File content is protobuf-serialized CompactBlock

Dart writes the block files, Rust reads them during scanning. The flow:

Dart: gRPC download -> CompactBlock.writeToBuffer() -> write file
Dart: call Rust write_block_metadata() -> validates files exist, writes to SQLite
Rust: scan_cached_blocks() -> FsBlockDb reads files -> prost::decode -> trial decrypt

5.3 Trial Decryption & Note Discovery

scan_cached_blocks() from zcash_client_backend performs trial decryption on every shielded output in the cached blocks:

  • For each Sapling output: attempts decryption with the wallet's Sapling IVK
  • For each Orchard action: attempts decryption with the wallet's Orchard IVK
  • Successful decryption = note belongs to this wallet → stored in wallet DB
  • Nullifiers are tracked to detect when notes are spent

5.4 Progress Reporting

After each batch of 1000 blocks, Dart calls get_sync_status() which reads WalletSummary from the DB. This provides fully_scanned_height and chain_tip_height, from which percentage is computed. The SyncState is updated and the UI reflects the new progress.

7. Address System

7.1 Unified Address (Sapling + Orchard)

The primary address shown to users is a ZIP-316 Unified Address containing Sapling and Orchard receivers only (transparent is excluded). This matches zodl/Zashi behavior.

  • Mainnet prefix: u1...
  • Receivers: Orchard (Pallas curve) + Sapling (Jubjub curve)
  • Sending wallets automatically select the highest-privacy receiver they support

7.2 Transparent Address

A separate transparent-only address is available via getTransparentAddress(). This is a standard t-address (P2PKH, prefix t1 on mainnet) derived from the same seed at BIP-44 path m/44'/133'/0'.

7.3 Compatibility with zodl/Zashi

Same mnemonic will produce different addresses between our wallet and zodl. This is expected — zodl's Android SDK uses specific git revisions of orchard and sapling-crypto via [patch.crates-io] that differ from the crates.io published versions we use. The key derivation math produces different curve points. Both are valid Zcash addresses from the same seed.
